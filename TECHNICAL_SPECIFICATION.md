# Техническая спецификация проекта

## 1. Что это за проект

`Elevenlabs-API` — это Colab-based workflow для quality-first транскрибации длинных аудио- и видеофайлов в Google Docs.

Проект ориентирован в первую очередь на русскоязычные лекции, вебинары, созвоны, учебные записи и другой длинный контент, где важны:
- хорошее качество распознавания;
- устойчивость на длинных файлах;
- удобный финальный output сразу в Google Docs;
- защита от повторной траты кредитов;
- возможность масштабировать дальнейшую работу через GitHub, PR и code review.

Важно не путать: это не просто обёртка над API ElevenLabs, а целостный pipeline, который включает выбор источника, подготовку медиа, выбор провайдера, транскрибацию, постобработку, запись результата в Google Docs и работу с manifest.

## 2. Цель проекта

Цель проекта — получить устойчивый workflow для транскрибации длинных записей в Google Docs из Google Colab с ElevenLabs как основным провайдером и OpenAI как fallback-путём.

Фокус проекта:
- quality-first распознавание;
- стабильная работа с длинными файлами;
- удобный пользовательский результат в Google Docs;
- минимизация лишних ручных действий;
- инженерная пригодность для дальнейшего развития через GitHub/Codex.

## 3. Основной пользовательский сценарий

Базовый сценарий выглядит так:
1. пользователь выбирает источник: локальный файл, несколько файлов, Google Drive file или Google Drive folder;
2. если на вход подано видео, workflow извлекает аудио;
3. выбирается provider path;
4. файл отправляется на транскрибацию;
5. результат проходит merge / постобработку при необходимости;
6. итоговый текст сохраняется в Google Docs;
7. состояние обработки фиксируется в manifest, чтобы не тратить кредиты повторно.

## 4. Поддерживаемые режимы источника

Поддерживаются 4 режима:
- computer: single file;
- computer: multiple files;
- Google Drive: single file;
- Google Drive: folder.

## 5. Провайдерная стратегия

### Основной провайдер
- `ElevenLabs / scribe_v2`

Это основной quality-first и основной production-путь проекта.

### Fallback
- `OpenAI / gpt-4o-transcribe`

Используется как fallback, когда нужен альтернативный провайдер или тестирование второй ветки.

### Speaker-aware сценарий
- `OpenAI / gpt-4o-transcribe-diarize`

Используется для сценариев, где важно разделение по спикерам, особенно для созвонов.

## 6. Верхнеуровневая архитектура

Pipeline проекта:

```text
Input source
  -> Media preparation
  -> Provider selection
  -> Transcription
  -> Post-processing / merge
  -> Google Docs output
  -> Manifest update
```

### 6.1. Input layer
Отвечает за:
- выбор источника;
- различение локального и Google Drive режима;
- обработку одного файла, нескольких файлов или папки;
- определение типа входного медиа.

### 6.2. Media preparation layer
Отвечает за:
- извлечение аудио из видео;
- подготовку файлов к STT-провайдеру;
- нормализацию входного формата;
- при необходимости разбиение длинного контента на части.

### 6.3. Provider layer
Отвечает за:
- выбор провайдера;
- вызов ElevenLabs или OpenAI;
- разделение логики между обычной транскрибацией и diarization-сценарием.

### 6.4. Chunking and merge layer
Отвечает за:
- smart split больших файлов;
- silence-aware split;
- overlap-aware merge;
- сборку финального результата после chunking.

### 6.5. Output layer
Отвечает за:
- запись результата в Google Docs;
- chunked insert длинного текста;
- формирование финального пользовательского артефакта.

### 6.6. Manifest layer
Отвечает за:
- фиксацию состояния обработки;
- защиту от повторной оплаты/повторной транскрибации;
- import existing / backfill;
- связь между исходным медиа и уже созданным Google Doc.

## 7. Почему manifest — обязательная часть решения

Manifest здесь не косметика, а важный системный слой. Он нужен, чтобы:
- не тратить кредиты повторно на уже обработанные файлы;
- не терять связь между входным медиа и созданным Google Doc;
- поддерживать import existing / backfill;
- восстанавливать историю прогонов.

Без manifest workflow деградирует в дорогой и плохо управляемый набор ручных действий.

## 8. Output-first принцип

Проект строится не вокруг идеи “просто получить transcript”, а вокруг идеи “получить пригодный конечный результат в Google Docs”.

Из этого следуют решения:
- финальный результат сохраняется только в Google Docs;
- локальные transcript-файлы не считаются главным конечным артефактом;
- JSON и другие технические выходы вторичны;
- логика pipeline подчинена надёжному сохранению результата в Google Docs.

## 9. Основные возможности

Подтверждённые и целевые возможности проекта:
- работа из Google Colab;
- поддержка Google Drive как источника;
- сохранение результата только в Google Docs;
- выбор output folder;
- несколько source modes;
- conflict modes;
- chunked insert в Google Docs;
- manifest со статусами обработки;
- import existing / backfill;
- поддержка keyterms;
- optional speaker split.

## 10. Секреты и безопасная модель работы

### Базовый принцип
Секреты не должны храниться:
- в `.py` файлах;
- в `.ipynb` outputs;
- в `README.md`;
- в документации;
- в git history.

### Основные секреты
Обязательный:
- `ELEVENLABS_API_KEY`

Опциональный:
- `OPENAI_API_KEY` — нужен только для OpenAI fallback или OpenAI diarization.

### Дополнительные credentials
Если для интеграции с Google Drive / Google Docs нужны дополнительные credentials, они тоже не должны коммититься в репозиторий.

### Рекомендуемый паттерн доступа
```python
from google.colab import userdata

ELEVENLABS_API_KEY = userdata.get("ELEVENLABS_API_KEY")
OPENAI_API_KEY = userdata.get("OPENAI_API_KEY")
```

### Важное поведение системы
Если пользователь не выбрал OpenAI-путь, отсутствие `OPENAI_API_KEY` не должно ломать запуск ElevenLabs-only сценария.

## 11. Практический runbook

### 11.1. Перед запуском
Нужно определить:
- source mode;
- тип входного файла: аудио или видео;
- нужен ли batch;
- нужен ли import existing / backfill;
- какой provider path нужен реально.

### 11.2. Базовая логика выбора провайдера
По умолчанию:
- основной путь — ElevenLabs;
- OpenAI — fallback;
- OpenAI diarize — отдельный сценарий для speaker-aware кейсов.

Если нет явной причины идти в OpenAI, основной путь — ElevenLabs.

### 11.3. Успешный прогон считается успешным только если
1. транскрибация завершилась;
2. результат корректно собран после split/merge при необходимости;
3. документ создан или обновлён в Google Docs;
4. manifest сохранил состояние, пригодное для повторного использования.

STT-ответ сам по себе ещё не означает продуктовый успех.

### 11.4. Что проверять после запуска
Нужно проверить:
- документ реально создался или обновился;
- текст выглядит целостным;
- manifest обновился;
- повторный запуск не должен без причины повторно тратить кредиты.

## 12. Основные риск-зоны и ограничения

### 12.1. OpenAI fallback не считается fully validated production path
OpenAI fallback встроен в архитектуру, но не должен автоматически считаться полностью production-ready только по факту наличия такой ветки.

Это особенно важно для:
- длинных файлов;
- speaker-aware сценариев;
- diarization;
- chunking + merge.

### 12.2. Ограничения OpenAI diarization + chunking
Когда diarization сочетается с разбиением длинного контента на части, возможны:
- потеря целостности speaker segmentation;
- ухудшение качества стыковки чанков;
- артефакты на границах частей;
- несогласованность speaker labels между chunk-результатами.

Даже технически успешная обработка отдельных чанков не гарантирует хороший итоговый продуктовый результат.

### 12.3. Ограничения длинных файлов в целом
Даже вне diarization длинные файлы остаются риск-зоной, потому что требуют:
- корректного split;
- аккуратного merge;
- контроля за потерями на стыках;
- проверки итоговой связности результата.

“Файл обработался без падения” не равно “файл обработался качественно”.

### 12.4. Ограничение private repo как меры безопасности
Private repo полезен, но не защищает от:
- случайного коммита секретов;
- утечки через notebook outputs;
- попадания credential-файлов в историю git;
- плохой дисциплины работы с runtime-артефактами.

### 12.5. Ограничение Colab-only модели
Если реальная жизнь проекта продолжит существовать только в Colab, а GitHub останется только обвязкой, то:
- code review будет частично формальным;
- PR workflow останется неполным;
- Codex будет работать по неполной картине;
- инженерная история проекта останется фрагментированной.

## 13. Что считается source of truth

На текущем этапе source of truth по логике проекта — это реальный рабочий код и его фактическое поведение на прогонах.

Документация нужна, чтобы:
- фиксировать архитектуру;
- не терять ограничения и known issues;
- ускорять вход нового помощника, ревьюера или нового чата;
- не таскать каждый раз весь handoff из переписки.

Если код и документация расходятся, приоритет у кода и реального поведения, но документацию нужно актуализировать как можно быстрее.
