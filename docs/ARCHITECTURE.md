# Architecture

## Архитектурная идея

Проект построен как end-to-end workflow, а не как один изолированный вызов speech-to-text API.

Его задача — взять длинный аудио- или видеоисточник, безопасно прогнать его через подходящий STT-путь, обработать ограничения провайдера и сохранить финальный результат в Google Docs с минимальными лишними действиями со стороны пользователя.

## Верхнеуровневый pipeline

```text
Input source
  -> Media preparation
  -> Provider selection
  -> Transcription
  -> Post-processing / merge
  -> Google Docs output
  -> Manifest update
```

## Основные модули логики

### 1. Input layer
Зона ответственности:
- выбор источника;
- различение локального и Google Drive режима;
- работа с одним файлом или папкой;
- определение типа входного медиа.

Ожидаемые source modes:
- computer: single file;
- computer: multiple files;
- Google Drive: single file;
- Google Drive: folder.

### 2. Media preparation layer
Зона ответственности:
- извлечение аудио из видео;
- базовая подготовка файлов к отправке в STT-провайдер;
- нормализация входного формата;
- при необходимости разбиение длинного контента на части.

Важно: на длинных файлах именно здесь часто возникает компромисс между качеством, устойчивостью и стоимостью.

### 3. Provider layer
Зона ответственности:
- выбор провайдера;
- передача нужных параметров в ElevenLabs или OpenAI;
- ветвление логики для обычной транскрибации и diarization-сценариев.

#### Primary path
- ElevenLabs / `scribe_v2`

#### Fallback path
- OpenAI / `gpt-4o-transcribe`

#### Speaker-aware path
- OpenAI / `gpt-4o-transcribe-diarize`

### 4. Chunking and merge layer
Зона ответственности:
- smart split больших файлов;
- overlap-aware merge;
- silence-aware split;
- объединение результата после chunking.

Это одна из самых чувствительных зон проекта, потому что именно тут чаще всего появляются потери контекста, дубли или артефакты на стыках чанков.

### 5. Output layer
Зона ответственности:
- финальная запись результата в Google Docs;
- chunked insert в документ при длинном тексте;
- поддержание стабильного итогового пользовательского формата.

Принципиально важно: финальный целевой output здесь — Google Docs.

### 6. Manifest layer
Зона ответственности:
- фиксация состояния обработки;
- защита от повторной оплаты/повторной транскрибации;
- import existing / backfill;
- повторное использование уже известных результатов.

## Почему manifest — часть архитектуры, а не “дополнительная мелочь”

Для этого проекта manifest — это не косметика. Он решает сразу несколько продуктовых проблем:
- не тратить кредиты повторно на уже обработанные файлы;
- не терять связь между входным медиа и уже созданным Google Doc;
- позволять аккуратно пересобирать историю прогонов;
- поддерживать import existing/backfill.

Без manifest workflow деградирует в дорогой и плохо управляемый набор ручных действий.

## Логика выбора провайдера

На уровне архитектуры важно фиксировать не только список провайдеров, но и приоритеты:

### ElevenLabs как default path
Причины:
- quality-first позиционирование;
- основной production-сценарий проекта;
- текущая ставка на качество распознавания.

### OpenAI как fallback
Причины:
- резервный путь;
- альтернативный сценарий;
- поддержка speaker-aware кейсов через diarize-модель.

Но OpenAI не должен считаться полностью production-validated только по факту наличия кода.

## Output-first principle

Важный архитектурный принцип проекта:

Проект строится не вокруг “получить transcript любой ценой”, а вокруг “получить пригодный конечный результат в Google Docs”.

Из этого следуют решения:
- локальные промежуточные артефакты не считаются финальным результатом;
- JSON и другие технические выходы вторичны;
- структура pipeline подчинена надёжному попаданию результата в Google Docs.

## Secrets model

Секреты не должны жить в коде и не должны коммититься в репозиторий.

Ожидаемая модель:
- хранение ключей в Google Colab Secrets / userdata;
- условное обращение к `OPENAI_API_KEY`, чтобы отсутствие ключа не ломало ElevenLabs-only путь;
- отсутствие hardcoded credentials в `.py`, `.ipynb`, README и docs.

## Архитектурные риски

### 1. Разрыв между Colab и GitHub
Если логика продолжает жить только в ноутбуке, а GitHub содержит только README и docs, репозиторий не становится настоящим source of truth.

### 2. Переоценка OpenAI fallback
Наличие fallback-ветки не равно её полной валидированности.

### 3. Слабая документированность ограничений
Если known limitations не вынесены отдельно, следующий помощник или ревьюер будет заново наступать на уже известные проблемы.

### 4. Смешение runtime-артефактов с исходниками
Если не держать репозиторий чистым, в git легко начинают попадать медиа, manifests, outputs и credential-следы.

## Архитектурная цель ближайшего этапа

На ближайшем этапе репозиторий должен перейти из состояния “документационная обвязка вокруг Colab-скрипта” в состояние “рабочая инженерная база проекта”, где:
- есть канонический `.py` файл для review;
- при необходимости есть `.ipynb` как Colab shell;
- документация описывает реальную архитектуру;
- known limitations и setup зафиксированы явно.
