# Архитектура и стратегия провайдеров

## Общая архитектурная идея

Проект строится как workflow с несколькими слоями:

1. **Входной слой** — выбор источника и подготовка файла.
2. **Media-prep слой** — извлечение аудио из видео и подготовка к транскрибации.
3. **STT-слой** — выбор и вызов провайдера транскрибации.
4. **Post-processing слой** — merge чанков, приведение результата в пригодный для чтения вид.
5. **Persistence/output слой** — запись результата в Google Docs и фиксация состояния в manifest.

## Почему основной провайдер — ElevenLabs

На текущем этапе основной quality-first путь проекта — `ElevenLabs / scribe_v2`.

Причины этого выбора:
- проект ориентирован на качество текста;
- это основная, предпочтительная ветка по текущему handoff;
- OpenAI пока рассматривается как fallback, а не как основной production-path.

## Роль OpenAI

OpenAI в проекте нужен не как замена всей архитектуры, а как fallback-провайдер.

Зафиксированные модели:
- `gpt-4o-transcribe`
- `gpt-4o-transcribe-diarize`

Использование OpenAI особенно важно там, где нужны:
- альтернативный STT-путь;
- сценарии со спикерами;
- дополнительная гибкость при развитии pipeline.

## Важная граница между supported и validated

Архитектурно OpenAI-ветка уже включена в проект.

Но нужно различать:
- **supported** — ветка добавлена и предусмотрена логикой проекта;
- **validated** — ветка реально подтверждена end-to-end прогонами на практических кейсах.

Для текущего проекта нельзя автоматически считать все OpenAI-сценарии fully validated.

## Chunking и merge

Для длинных файлов архитектура должна учитывать ограничения провайдера и размера входных файлов.

По handoff в проектной логике уже фигурируют:
- smart split;
- silence-aware split;
- overlap-aware merge.

Это означает, что проектный pipeline должен не только “резать файл”, но и:
- делать разбиение аккуратно;
- избегать потери текста на стыках;
- собирать текст обратно без заметных артефактов.

## Speaker split

Speaker split — полезная, но чувствительная часть архитектуры.

Практический смысл:
- для лекций он может быть вторичен;
- для созвонов, интервью и дискуссий — критически важен.

При этом именно здесь обычно выше риск проблем в длинных файлах, чанкинге и согласованности результата между кусками.

## Manifest как архитектурный слой

Manifest — это не просто технический файл.

В контексте проекта он выполняет роль операционного слоя, который:
- связывает исходный файл и итоговый Google Doc;
- хранит статус обработки;
- позволяет импортировать уже существующие результаты;
- уменьшает риск повторной траты API-кредитов.

## Практический архитектурный вывод

У проекта уже есть не просто “скрипт для транскрибации”, а каркас pipeline.

Поэтому дальнейшее развитие в GitHub должно идти не как хаотичные правки ноутбука, а как развитие отдельных смысловых блоков:
- input/source handling;
- media prep;
- provider routing;
- chunking/merge;
- Google Docs output;
- manifest logic;
- error handling;
- reviewability и тестируемость.
