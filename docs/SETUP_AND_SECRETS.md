# Setup and Secrets

## Цель этого документа

Этот файл фиксирует минимально безопасную модель настройки проекта для работы в Google Colab без захардкоженных секретов и без случайной утечки чувствительных данных в репозиторий.

## Базовый принцип

Секреты не должны храниться:
- в `.py` файлах;
- в `.ipynb` outputs;
- в `README.md`;
- в документации;
- в git history.

Рабочая модель проекта:
- код хранится в репозитории;
- секреты хранятся в Colab Secrets / `userdata`;
- код читает секреты только в runtime.

## Основные секреты

### Обязательный
- `ELEVENLABS_API_KEY`

Нужен для основного production-пути через ElevenLabs.

### Опциональный
- `OPENAI_API_KEY`

Нужен только если пользователь реально выбирает OpenAI fallback или сценарий с diarization через OpenAI.

Важно: отсутствие `OPENAI_API_KEY` не должно ломать ноутбук, если выбран путь только через ElevenLabs.

### Дополнительные Google-related credentials
В зависимости от реализации workflow могут понадобиться и другие Google-related настройки или credentials для интеграции с Drive / Docs. Их тоже нельзя хранить в коде или коммитить в репозиторий.

## Рекомендуемый паттерн доступа к секретам

```python
from google.colab import userdata

ELEVENLABS_API_KEY = userdata.get("ELEVENLABS_API_KEY")
OPENAI_API_KEY = userdata.get("OPENAI_API_KEY")
```

## Что важно проверить перед коммитом

Перед загрузкой кода в GitHub нужно проверять не только сам исходник, но и Colab-артефакты.

### Для `.py`
Проверить, что в коде нет:
- реальных ключей;
- credential JSON;
- временных debug-констант;
- локальных путей, которые раскрывают лишний контекст.

### Для `.ipynb`
Проверить, что в ноутбуке нет:
- outputs с реальными ответами API;
- засвеченных значений секретов;
- логов с приватными ссылками;
- временных данных по Google Docs / Drive, которые не должны храниться в git.

## Практическое правило для Colab notebooks

Если в репозиторий загружается `.ipynb`, желательно перед коммитом:
- очистить outputs;
- убрать лишние экспериментальные ячейки;
- убедиться, что notebook не содержит секретов, токенов и чувствительных runtime-артефактов.

## Чего нельзя делать

### Нельзя хранить ключи прямо в коде
Плохо:

```python
ELEVENLABS_API_KEY = "real_key_here"
```

### Нельзя полагаться на “private repo всё спишет”
Даже private repo не является оправданием для коммита секретов.

### Нельзя смешивать secrets и docs
Документация должна объяснять, какие секреты нужны и как они используются, но не содержать сами значения.

## Минимальный setup checklist

### Для запуска ElevenLabs-only пути
1. настроить `ELEVENLABS_API_KEY` в Colab Secrets;
2. убедиться, что notebook/скрипт умеет читать `userdata`;
3. не задавать OpenAI путь по умолчанию, если `OPENAI_API_KEY` отсутствует.

### Для запуска OpenAI fallback
1. дополнительно настроить `OPENAI_API_KEY`;
2. проверить, что fallback включается только при реальном выборе соответствующего режима;
3. отдельно помнить, что наличие ключа ещё не означает, что ветка fully validated.

## Что желательно документировать рядом с кодом

Когда в репозиторий будет загружен основной `.py`, рядом с ним полезно явно зафиксировать:
- какие секреты обязательны;
- какие опциональны;
- какой режим без каких ключей должен работать;
- какие ветки считаются экспериментальными.

## Риск, который нельзя недооценивать

Самый частый реальный риск в подобных проектах — не прямой hardcode ключа, а случайная утечка через notebook output, временный debug print, промежуточный JSON или нечищенный runtime-артефакт.

Поэтому безопасная работа здесь — это не только “не вставлять ключ в код”, но и регулярно проверять весь коммит как артефакт целиком.
