# Project Overview

## Что это за проект

`Elevenlabs-API` — это Colab-based workflow для quality-first транскрибации длинных аудио- и видеофайлов в Google Docs.

Проект ориентирован в первую очередь на русскоязычные лекции, вебинары, созвоны, образовательные записи и другой длинный контент, где важны:
- хорошее качество распознавания;
- устойчивость на длинных файлах;
- удобный output сразу в Google Docs;
- защита от повторной траты кредитов;
- возможность безопасно масштабировать workflow.

## Основной сценарий

Базовый пользовательский сценарий выглядит так:
1. пользователь выбирает источник: локальный файл, несколько файлов, Google Drive file или Google Drive folder;
2. если на вход подано видео, из него извлекается аудио;
3. workflow отправляет файл на транскрибацию через выбранного провайдера;
4. результат нормализуется и сохраняется в Google Docs;
5. информация о прогоне и результате записывается в manifest;
6. при повторном запуске workflow может использовать manifest, чтобы не тратить кредиты повторно.

## Основная стратегия провайдеров

### Primary provider
- `ElevenLabs / scribe_v2`

Это основной quality-first путь, который рассматривается как главный production-сценарий.

### Fallback provider
- `OpenAI / gpt-4o-transcribe`

Используется как fallback, когда нужен альтернативный провайдер или тестирование второй ветки.

### Diarization scenario
- `OpenAI / gpt-4o-transcribe-diarize`

Используется для сценариев, где важно разделение по спикерам, особенно для созвонов. При этом именно здесь остаются основные известные ограничения, особенно в сочетании с chunking на длинных файлах.

## Почему проект живёт в GitHub

Изначально workflow развивался в Google Colab, но для дальнейшей работы этого уже недостаточно.

GitHub нужен не ради формальности, а как рабочая база для:
- прозрачной истории изменений;
- code review;
- PR workflow;
- подключения Codex и других агентных помощников;
- нормальной технической документации по проекту.

## Что считается source of truth

На текущем этапе важно различать два уровня истины:

### 1. Логика проекта
Source of truth по логике проекта — реальный рабочий код и его фактическое поведение на прогонах.

### 2. Документация и handoff-контекст
Документация в репозитории фиксирует:
- архитектурные решения;
- ограничения;
- expected flow;
- known issues;
- правила безопасной работы с секретами и артефактами.

Если документация расходится с реальным кодом, приоритет у кода и реального поведения, но документацию нужно обновить как можно быстрее.

## Что важно не путать

### Это не просто “API-обёртка”
Несмотря на имя репозитория, проект — это не только работа с API ElevenLabs. Это целостный workflow:
- выбор источника;
- обработка аудио/видео;
- вызов провайдера;
- постобработка;
- загрузка в Google Docs;
- работа с manifest.

### Это не generic transcript exporter
Финальный целевой output здесь — именно Google Docs, а не просто текстовый файл, JSON или локальный артефакт.

### Это не fully validated multi-provider engine
ElevenLabs-путь рассматривается как основной и наиболее надёжный. OpenAI fallback встроен архитектурно, но не все его ветки подтверждены полными реальными end-to-end прогонами.

## Текущий практический вывод

Документация должна помогать в двух сценариях:
1. быстрый вход в проект для нового чата / нового помощника / ревьюера;
2. сохранение накопленного контекста между итерациями, чтобы не тащить весь handoff заново в каждую новую ветку.
