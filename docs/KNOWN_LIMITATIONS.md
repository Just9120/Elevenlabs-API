# Known Limitations

## Зачем нужен этот документ

Для этого проекта особенно важно отдельно фиксировать не только возможности, но и ограничения.

Без такого файла очень легко начать переоценивать готовность fallback-веток, забыть уже известные технические компромиссы и заново тратить время на повторное выяснение проблем, которые уже были замечены раньше.

## Главное ограничение на текущем этапе

### OpenAI fallback не считается fully validated production path

OpenAI fallback встроен в архитектуру проекта, но его нельзя автоматически считать полностью проверенным production-решением только потому, что соответствующий код уже был когда-то добавлен.

Это особенно важно для:
- длинных файлов;
- speaker-aware сценариев;
- режима diarization;
- chunking + merge цепочки.

## Ограничения OpenAI diarization + chunking

Это одна из основных известных риск-зон проекта.

Когда diarization сочетается с разбиением длинного контента на части, возможны проблемы такого класса:
- потеря целостности speaker segmentation;
- ухудшение качества стыковки чанков;
- артефакты на границах частей;
- несогласованность speaker labels между разными chunk-результатами.

Даже если отдельные куски распознаются технически успешно, итоговый собранный результат может быть хуже ожидаемого по качеству как продуктовый артефакт.

## Ограничения длинных файлов в целом

Даже вне diarization длинные файлы остаются сложной зоной, потому что требуют:
- корректного split;
- аккуратного merge;
- контроля за потерями на стыках;
- проверки итоговой связности результата.

То есть “длинный файл обработался без падения” ещё не равно “длинный файл обработался качественно”.

## Ограничение по source of truth

Сейчас GitHub-репозиторий ещё не должен восприниматься как полностью завершённый инженерный source of truth, пока в нём не загружен и не подтверждён основной рабочий кодовый артефакт.

На текущем этапе репозиторий уже содержит документацию и инфраструктурную базу, но это не то же самое, что полностью оформленная кодовая база проекта.

## Ограничение по README

README сам по себе недостаточен для этого проекта.

Причина:
- проект слишком процессный;
- в нём есть архитектурные развилки;
- есть operational логика;
- есть known limitations, которые нельзя прятать внутрь общего обзорного текста.

Именно поэтому документация вынесена в отдельные docs-файлы.

## Ограничение private repo как меры безопасности

Private repo — это полезно, но этого недостаточно.

Он не защищает от:
- случайного коммита секретов;
- утечки через notebook outputs;
- попадания credential-файлов в историю git;
- плохой дисциплины обращения с runtime-артефактами.

## Ограничение Colab-only режима разработки

Если реальная жизнь проекта продолжит существовать только в Colab, а GitHub будет содержать только docs и обвязку, то:
- code review останется частично формальным;
- PR workflow будет неполным;
- Codex будет работать на неполной картине;
- инженерная история проекта останется фрагментированной.

Поэтому следующий зрелый шаг — перенос канонического `.py` в репозиторий.

## Ограничение notebook-as-main-source

Использовать один только `.ipynb` как главный артефакт для review неудобно, потому что:
- diffs менее удобны;
- обсуждать изменения сложнее;
- агентам и ревьюерам труднее работать с логикой;
- notebook чаще тащит лишние outputs и шум.

Поэтому практический приоритет — иметь `.py` как основной читаемый кодовый артефакт, а `.ipynb` держать как Colab shell при необходимости.

## Ограничение документации без ревизии после кодовых изменений

Даже хорошая документация быстро устаревает, если её не обновлять вместе с изменением логики pipeline.

Значит для этого проекта нужно считать нормой:
- менять docs вместе с важными кодовыми изменениями;
- отдельно фиксировать новые ограничения;
- не оставлять архитектурные сдвиги только в PR-обсуждении без обновления docs.

## Практический вывод

На текущем этапе зрелое и честное позиционирование проекта такое:
- основной production-путь: ElevenLabs;
- OpenAI fallback: полезен, но не fully validated;
- diarization + chunking: зона повышенного риска;
- GitHub: уже правильная база для масштабирования, но ещё не завершён как полноценный кодовый source of truth;
- docs нужны не как украшение, а как защита от потери контекста и повторения уже известных ошибок.
