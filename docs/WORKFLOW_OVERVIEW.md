# Обзор workflow

## Цель workflow

Workflow нужен для quality-first транскрибации длинных записей в Google Docs.

Приоритеты проекта:
- качество текста выше скорости;
- сохранение результата в Google Docs — основной целевой output;
- защита от повторной платной обработки;
- возможность масштабировать работу через GitHub и review.

## Базовый сценарий

Упрощённо поток выглядит так:

1. Пользователь выбирает источник файла.
2. Система определяет, это аудио или видео.
3. Если это видео, из него извлекается аудио.
4. Пользователь выбирает провайдера транскрибации.
5. При необходимости включаются дополнительные режимы: язык, speaker split и т.д.
6. Если файл слишком большой для ограничений провайдера, применяется безопасное разбиение.
7. Частичные результаты собираются обратно в единый текст.
8. Итог вставляется в Google Docs чанками.
9. Информация о результате и состоянии обработки фиксируется в manifest.
10. Повторные запуски используют import existing / backfill, где это возможно, чтобы не тратить кредиты повторно.

## Поддерживаемые source modes

По handoff зафиксированы 4 режима:

- компьютер: один файл;
- компьютер: несколько файлов;
- Google Drive: один файл;
- Google Drive: папка.

## Целевой output

Финальный результат проекта:
- Google Docs.

Что не считается основным финальным output:
- локальные transcript-файлы;
- JSON-экспорты;
- временные промежуточные артефакты.

## Почему workflow строится вокруг Google Docs

Google Docs выбран как основной финальный слой, потому что он лучше подходит для:
- чтения длинных транскриптов;
- ручной правки;
- совместной работы;
- хранения ссылок на итоговые документы;
- дальнейшего переноса в базу знаний.

## Manifest и защита от повторной обработки

Один из ключевых практических элементов workflow — manifest.

Он нужен, чтобы:
- отслеживать, что уже было обработано;
- избегать повторной траты кредитов;
- поддерживать import existing / backfill;
- служить операционным слоем между запуском и итоговыми документами.

## Что важно держать в голове

Этот workflow — не просто “нотбук, который делает STT”.

Это заготовка под устойчивый pipeline, в котором есть:
- выбор источника;
- обработка медиа;
- провайдерная логика;
- ограничения моделей;
- итоговый экспорт в Google Docs;
- учёт ранее обработанных результатов;
- перспектива нормального GitHub / PR / review процесса.
