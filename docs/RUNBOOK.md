# Runbook

## Зачем нужен этот документ

README описывает проект в целом, но для реальной работы нужен более прикладной документ: как мыслить про запуск workflow, какие режимы существуют и на что смотреть при сбоях.

Этот runbook — не замена коду, а практическая карта эксплуатации.

## Базовый операционный сценарий

### 1. Выбрать источник
Поддерживаемые режимы:
- компьютер: один файл;
- компьютер: несколько файлов;
- Google Drive: один файл;
- Google Drive: папка.

На этом этапе важно сразу понимать:
- что именно подаётся на вход;
- это аудио или видео;
- требуется ли batch-обработка;
- нужен ли import existing/backfill вместо новой транскрибации.

### 2. Определить провайдерный путь
Основная логика выбора:
- по умолчанию — ElevenLabs;
- OpenAI — fallback;
- OpenAI diarize — отдельный сценарий для speaker-aware задач.

Практический принцип:
если нет явной причины идти в OpenAI, основной путь — ElevenLabs.

### 3. Проверить секреты
Минимум нужно убедиться, что:
- `ELEVENLABS_API_KEY` доступен для ElevenLabs-пути;
- `OPENAI_API_KEY` задан только если реально нужен OpenAI-режим.

Если пользователь не выбрал OpenAI, отсутствие `OPENAI_API_KEY` не должно ломать запуск.

### 4. Подготовить входной файл
Если на входе видео, workflow должен корректно извлечь аудио.

Если файл длинный, нужно заранее помнить, что возможны:
- chunking;
- merge после chunking;
- ограничения по diarization-сценариям.

### 5. Выполнить транскрибацию
Во время запуска важно понимать, что есть два уровня успеха:

#### Технический успех
- pipeline не упал;
- провайдер вернул данные;
- результат дошёл до этапа постобработки.

#### Продуктовый успех
- итоговый текст реально сохранён в Google Docs;
- manifest обновлён;
- пользователь может повторно найти и использовать результат без повторной траты кредитов.

## Как интерпретировать “успешный прогон”

Прогон нельзя считать по-настоящему успешным только потому, что STT API вернул текст.

Успешный прогон здесь — это когда:
1. транскрипция завершилась;
2. результат корректно собран после merge/chunking при необходимости;
3. документ создан или обновлён в Google Docs;
4. manifest сохранил состояние, пригодное для повторного использования.

## Практические сценарии использования

### Сценарий A. Одна длинная лекция
Оптимальный путь:
- один источник;
- ElevenLabs;
- сохранение в Google Docs;
- запись в manifest.

### Сценарий B. Папка файлов из Google Drive
Оптимальный путь:
- Google Drive folder mode;
- последовательная обработка;
- контроль конфликтов;
- manifest как защита от дублей.

### Сценарий C. Созвон с разными спикерами
Возможный путь:
- OpenAI diarize;
- повышенное внимание к ограничениям chunking;
- повышенная осторожность при трактовке результата как production-ready.

### Сценарий D. Import existing / backfill
Используется, когда уже есть созданные результаты и их нужно аккуратно завести в manifest без повторной оплаты транскрибации.

## Что проверять при проблемах

### Проблема: workflow падает при отсутствии OpenAI key
Что это значит:
- нарушен принцип optional fallback;
- OpenAI-путь слишком жёстко завязан на общий startup flow.

Что считать правильным поведением:
- если выбран только ElevenLabs, отсутствие `OPENAI_API_KEY` не должно ломать запуск.

### Проблема: файл слишком большой
Что это значит:
- нужно корректно отрабатывать smart split;
- важно потом правильно собрать результат через merge.

### Проблема: плохой результат на speaker-aware кейсе
Что это значит:
- возможно, ограничение связано не с общей логикой pipeline, а именно с OpenAI diarization + chunking.

### Проблема: результат есть, но Google Doc не создан
Что это значит:
- техническая транскрибация не завершила продуктовый pipeline;
- нужно искать проблему в output layer, а не в распознавании.

### Проблема: повторно тратятся кредиты
Что это значит:
- manifest не используется корректно;
- import existing/backfill работает не так, как ожидалось;
- идентификация уже обработанного файла недостаточно надёжна.

## Operational checklist перед запуском

### Минимум
- понятен source mode;
- выбран корректный provider;
- проверены нужные secrets;
- нет ожидания, что OpenAI fallback автоматически равен production-grade пути.

### Желательно
- понятен expected output document;
- понятна стратегия конфликтов;
- заранее ясно, нужен ли manifest import/backfill.

## Operational checklist после запуска

Нужно проверить:
- документ реально создался или обновился;
- текст выглядит целостным;
- manifest сохранил состояние;
- повторный запуск не должен без причины заново оплачивать уже обработанный файл.

## Что важно для GitHub/Codex workflow

Когда основной `.py` файл появится в репозитории, этот runbook должен использоваться как прикладной контекст для:
- code review;
- smoke-test мышления;
- обсуждения багов;
- подготовки PR, которые меняют логику pipeline.

То есть вопрос в PR должен звучать не только “код компилируется или нет”, а “не ломает ли изменение реальный рабочий сценарий транскрибации и output в Google Docs”.
